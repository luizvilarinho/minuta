## Findings

### youtube.rs

**[importante] `resolve_yt_dlp`: nome de binário no branch `#[cfg(not(windows))]` está errado**

```rust
} else {
    "yt-dlp-x86_64-pc-windows-msvc"   // ← sem .exe, mas ainda com triple Windows
};
```

O branch `not(windows)` usa o triple `x86_64-pc-windows-msvc` sem extensão. Num Linux/macOS isso buscaria um arquivo com nome de triple Windows. O correto seria o triple do host de compilação (ex: `x86_64-unknown-linux-gnu`). Por ora o app é Windows-only (binário na pasta `binaries/` é `.exe`), então não quebra, mas o código transmite intenção errada e falharia silenciosamente num port futuro.

---

**[importante] `transcribe_path` duplica lógica de `transcribe_audio` sem DRY**

`transcribe_path` (nova função pública) reproduz exatamente o mesmo fluxo de `transcribe_audio` (validação de tamanho, ffmpeg, split, chunks) sem reutilizá-lo. A única diferença real é que não usa `TranscribeState` (sem resume). Isso cria dois lugares para manter o mesmo pipeline. O correto seria `transcribe_audio` chamar `transcribe_path` e aplicar o `TranscribeState` em volta, ou extrair as etapas intermediárias compartilhadas.

---

**[nit] `extract_video_id` é chamada e resultado descartado antes do download**

```rust
extract_video_id(&url)?;   // valida, mas o id extraído é jogado fora
// ...
.args([..., &url_clone])   // passa a URL bruta ao yt-dlp
```

O `video_id` extraído não é usado — o yt-dlp recebe a URL completa. A validação é útil (rejeita lixo antes de invocar o processo), mas o código poderia ser mais explícito: ou usa o ID, ou a função se chama `validate_youtube_url`. Sem impacto funcional, mas comunica mal a intenção.

---

**[nit] Stubs `youtube_is_connected`, `youtube_login`, `youtube_disconnect` retornam hardcoded `true`/`Ok(())`**

Os três comandos são no-ops. Isso faz a UI de "Conectar conta do YouTube" nas Settings nunca mostrar estado desconectado de forma persistente — `youtube_is_connected` sempre retorna `true`. O componente `YoutubeUrlInput` tem um caminho de código para `!ytConnected` que nunca será exibido. Aceitável se a autenticação OAuth ainda não está implementada, mas deve ser documentado como débito técnico.

---

### Settings.tsx

O restante da UI foi migrado para inglês, mas as strings da seção YouTube em `Settings.tsx` ficaram em português:

- `"YouTube conectado"`
- `"Desconectar"`
- `"Aguardando login…"`
- `"Conectar conta do YouTube"`

Isso quebra a consistência visual da interface.

**[nit] Prop `onTextLoaded` declarada na interface mas não desestruturada/usada no componente**

```tsx
interface YoutubeUrlInputProps {
  onTextLoaded: (text: string) => void;  // declarada
  ...
}

export function YoutubeUrlInput({
  onTranscriptionStart,
  onTranscriptionDone,
  onTranscriptionError,
  isLoading,
  ytConnected,
  transcriptionModel,
  // onTextLoaded ausente na desestruturação
}: YoutubeUrlInputProps) {
```

A prop existe na interface e é passada de `App.tsx`, mas não é extraída nem utilizada internamente. Sem consequência funcional (a URL YouTube sempre gera transcrição via IA, nunca texto pré-existente), mas a interface mente sobre o contrato.

### SummaryControls.tsx

**[nit] `isPromptExpanded` e `isAdditionalExpanded` são estados independentes que poderiam ser derivados de um único estado de "painel aberto"**

Quatro booleans de colapso (`isPromptSettingsExpanded`, `isPromptExpanded`, `isAdditionalExpanded`, `isSavedPromptsExpanded`) gerenciam hierarquias de accordions que poderiam ser colapsados num único `activePanel: string | null`. Baixo impacto, mas aumenta o surface de bugs de estado.

---

### SummaryReadView.tsx

**[nit] `renderMarkdown` trata linhas mistas em blocos de lista de forma ingênua**

```tsx
const isListBlock = lines.some((l) => l.trimStart().startsWith("- ") || ...)
if (isListBlock) {
  return (
    <ul>
      {lines.map((l, j) => <li key={j}>{l.trimStart().slice(2)}</li>)}
    </ul>
  );
}
```

Se um bloco contiver linhas mistas (texto de parágrafo seguido de item `- `), todos os itens do bloco são renderizados como `<li>`, incluindo o texto introdutório. Para o prompt padrão isso não é problema (o modelo gera listas limpas), mas é frágil para prompts customizados do usuário.

---

### App.tsx

**[nit] `ytConnected` sincronizado via `useEffect` em mount e no `onClose` do Settings, mas não no `setInputMode("youtube")`**

Se o usuário for em Settings, conectar o YouTube, fechar as Settings e já estar na aba YouTube, o estado é atualizado corretamente. Mas se o usuário abrir Settings, conectar, fechar e então mudar para o modo YouTube pela primeira vez, o fluxo funciona. O único caso não coberto é: mudar para o modo YouTube *antes* de abrir Settings (estado inicial hardcoded como `false`). Como `youtube_is_connected` sempre retorna `true`, isso não manifesta hoje, mas é um débito de consistência.

### Cargo.toml

**[nit] Feature `cookies` adicionada ao `reqwest` sem uso aparente**

```toml
reqwest = { version = "0.12", features = ["json", "multipart", "cookies"] }
```

Nenhum código nos arquivos modificados usa `cookie_store` ou cookie jar do reqwest. Se foi adicionado como preparação para o OAuth YouTube, deve ser comentado. Caso contrário, é dependência desnecessária que aumenta o binário final.
