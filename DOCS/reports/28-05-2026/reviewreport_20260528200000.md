## `src-tauri/src/commands/youtube.rs`

### 1. Correção

**Faz o que foi pedido?** Sim. Os três commands `youtube_is_connected`, `youtube_login`, `youtube_disconnect` estão implementados. `fetch_youtube_transcript` foi refatorada para usar o WebView2 com fetch autenticado. A lógica de fallback por idiomas e candidatos (json3/srv3 × asr) está presente.

**Deadlock de `cookies_for_url` no Windows:** O risco identificado pelo explorerAgent é real: `cookies_for_url` é síncrono e, em handlers síncronos do Tauri, pode deadlocks por executar callbacks na thread principal do WebView2. Os três commands são `async` e chamam `cookies_for_url` dentro de um contexto tokio — correto. O `youtube_login` usa `tokio::time::sleep` antes de cada chamada a `cookies_for_url`, o que mantém o código em contexto async. Sem problemas de deadlock detectados.

**`on_page_load` no builder (Tauri 2.11.2):** O código usa `.on_page_load(...)` como método do `WebviewWindowBuilder`, antes do `.build()`. Isso é exatamente o que está disponível na API do Tauri 2.x — o método existe no builder, não na janela construída. O codeAgent registrou a divergência da API e resolveu corretamente com `Arc<Mutex<Option<oneshot::Sender<()>>>>`.

**`eval_with_callback` com `oneshot` — risco de sender nunca chamado:** O callback do `eval_with_callback` pode não ser chamado se o WebView for destruído antes da execução do JS ou se o `eval` falhar silenciosamente após aceitar o call. O timeout de 15s cobre o caso nominal. Contudo, se o WebView for fechado (ex.: `youtube_disconnect` executado em paralelo enquanto `fetch_youtube_transcript` aguarda o receiver), o `RecvError` resulta em `Err("canal fechado")` — que é uma mensagem de erro aceitável. O risco real é o `eval_with_callback` nunca disparar o callback em caso de crash do WebView, o que resultaria no timeout de 15s sendo consumido por cada candidato. Com 4 candidatos × idiomas, isso pode causar uma espera longa sem mensagem clara de erro. `[nit]` — ver item de simplicidade abaixo.

**Bug `[crítico]` — `youtube_login` pode deixar o frontend travado:**
Quando o usuário já tem uma janela `yt-login` aberta, o código retorna `Ok(false)` imediatamente. Do ponto de vista do frontend, `invoke("youtube_login")` resolve com `false` e `setYtConnected(false)` é chamado — o botão sai do estado `ytLoading` e mostra "Conectar conta do YouTube" novamente, sem nenhum indicativo de que há uma janela de login já aberta aguardando ação. O usuário pode se confundir, mas o comportamento funcional está correto (a janela foi colocada em foco). Entretanto, a consequência é que o comando retorna **antes** do usuário concluir o login — o estado no frontend fica `false` mesmo enquanto o usuário está logando na janela. Isso é um problema de UX pontual, não um crash. Rebaixado para `[importante]`.

**`[importante]` — `youtube_login`: retorno prematuro quando janela já existe**
Se a janela `yt-login` já está aberta e o usuário clica novamente em "Conectar", o command retorna `Ok(false)` instantaneamente. O frontend atualiza `ytConnected = false` e `ytLoading = false`. A janela fica aberta sem que o estado seja atualizado quando o login for concluído (a segunda chamada ao command foi descartada). Após o usuário fechar a janela manualmente ou concluir o login, o frontend não saberá que está conectado até que o modal de Settings seja reaberto (que aciona `youtube_is_connected` no `useEffect`). Funcionalmente não quebra nada — o `youtube_is_connected` no reopen do Settings vai atualizar o estado — mas a UX é ruim: o usuário vê "Conectar" no modal mesmo tendo acabado de logar.

### 2. Simplicidade

**`[nit]` — `eprintln!` de debug em produção:** Há 3 chamadas `eprintln!` no código: no `fetch_available_languages` (html length, languages list) e no `fetch_youtube_transcript` (url sendo tentada, body len). O padrão em `summarize.rs` não usa nenhum debug logging. Esses `eprintln!` devem ser removidos antes do merge ou condicionados a `#[cfg(debug_assertions)]`.

**`[nit]` — `Url` importada como `tauri::webview::Url`:** Funciona, mas a importação de `Url` via re-export do Tauri é não-idiomática em Rust. A crate `url` já é dependência transitiva — se o projeto precisar usar `Url` em outro módulo, vai gerar confusão. Aceitável para um único arquivo, mas vale um comentário de atenção.

## `src-tauri/capabilities/youtube.json`

### 1. Correção

As permissões `core:default`, `core:window:allow-close`, `core:window:allow-set-focus`, `core:webview:allow-webview-show`, `core:webview:allow-webview-hide` estão incluídas. Para Tauri 2.x, `core:default` inclui as permissões básicas de IPC. As janelas `yt-login` e `yt-fetch` precisam conseguir navegar para URLs externas — isso é controlado pela CSP e pela declaração `WebviewUrl::External`, não pela capability em si. A capability cobre o necessário para as operações realizadas pelos commands (`close`, `set_focus`, `show`/`hide`).

**`[importante]` — permissão `core:webview:allow-cookies` ausente:** O command `youtube_is_connected` e `youtube_login` chamam `win.cookies_for_url(...)`. Em Tauri 2.x, operações de cookie sobre o WebView podem requerer `core:webview:allow-cookies` ou equivalente na capability da janela. O `cargo check` passou, mas `cargo check` não valida capabilities em runtime — a ausência dessa permissão pode causar erro silencioso (cookies retornam lista vazia) em builds release com `TAURI_SKIP_DEVSERVER_CHECK=false`. Verificar na documentação ou testar em build release antes do merge.

**`[importante]` — `clear_all_browsing_data` não tem permissão explícita:** O command `youtube_disconnect` chama `win.clear_all_browsing_data()`. Não há permissão `core:webview:allow-clear-all-browsing-data` ou similar na capability. Mesmo raciocínio do item anterior — pode falhar silenciosamente ou com erro em runtime.

### 2–6. Demais categorias

Estrutura idêntica ao `default.json` existente. Sem outros problemas.
