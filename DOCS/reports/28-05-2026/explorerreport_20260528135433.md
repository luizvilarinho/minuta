## Resumo Executivo

A funcionalidade de resumir vídeos do YouTube é **viável tecnicamente** com **dificuldade Média**, dependendo da abordagem escolhida. Existem dois caminhos viáveis:

1. **Abordagem Recomendada (Média dificuldade):** Usar a biblioteca Python youtube-transcript-api como processo filho no Tauri, extraindo apenas transcrições/legendas sem baixar áudio.
2. **Abordagem Alternativa (Alta dificuldade):** YouTube Data API v3 oficial com autenticação OAuth2 (requer API key e comply com ToS mais rigorosos).

A terceira opção — yt-dlp para baixar áudio — é possível mas **viola a constraint original** de não baixar arquivos de vídeo.

## Análise Detalhada

### 1. Viabilidade de Buscar Transcrições/Legendas do YouTube

#### 1.1 Soluções Disponíveis

**A. YouTube Transcript API (Python)**
- **Biblioteca:** youtube-transcript-api (PyPI)
- **Fonte Reputação:** High (562 code snippets no Context7)
- **Funcionamento:** Reverse-engineers o endpoint de transcrições do YouTube, extraindo legendas automáticas ou manuais sem requerer download do vídeo
- **Langs:** Suporta múltiplos idiomas (pode buscar transcrições em PT-BR)
- **No Tauri:** Pode ser executado como processo filho via std::process::Command
- **Vantages:** Simples, não requer API key do YouTube, funciona offline após extraction
- **Desvantages:** Não é uma API oficial — YouTube pode quebrar a integração a qualquer momento

**B. YouTube Data API v3 Oficial**
- **Endpoint:** captions.list() e captions.download()
- **Autenticação:** OAuth2 ou API Key
- **Funcionamento:** Oficial, documentado, com suporte do Google
- **Reqs:** Precisa de Google Developer Console account, YouTube Data API habilitada
- **Rate Limiting:** 10,000 units por dia (cada caption fetch conta como unidades)
- **ToS Compliance:** Claro, mas com restrições de distribuição
- **No Tauri:** Implementar client HTTP em Rust via eqwest (já no Cargo.toml)
- **Desvantages:** Setup mais complexo, requer credenciais Google no app

**C. yt-dlp (Command-line)**
- **Funcionamento:** Pode extrair apenas transcrições com flag --write-subs --skip-download
- **No Tauri:** Executar como processo filho (precisa yt-dlp instalado no sistema)
- **Vantages:** Robusto, suporta fallbacks, atualiza automaticamente
- **Desvantages:** Dependência externa, varia por SO, gray zone com YouTube ToS

#### 1.2 Crates Rust Disponíveis

Busca no crates.io retornou:
- yt-dlp (36.837 downloads) — Wrapper async ao redor do executável yt-dlp
- youtube (787 downloads) — Crate genérico, pouca cobertura

**Conclusão:** Não existe crate Rust consolidado que implemente reverse-engineering de transcrições. A opção mais prática é usar Python + subprocess.

### 3. Termos de Uso e Distribuição

#### 3.1 YouTube Data API v3 Oficial

**Compliance:**
- Permitido usar em apps desktop
- Requer divulgação clara de política de privacidade
- Usuário deve saber que a transcrição é baixada dos servidores do YouTube
- Não pode redistribuir transcrições sem consentimento

**Distribuição do App:**
- Sim, distribuível, desde que declare na política de privacidade
- Precisa documentar: "Este app usa YouTube API para buscar transcrições públicas de vídeos"
- Google pode revogar acesso em qualquer momento
- **Riscos Baixos** para distribuição, mas **setup administrativo maior**

#### 3.2 youtube-transcript-api (Reverse-engineering)

**Compliance:**
- Não é oficial
- YouTube ToS não explicitamente proíbe, mas está em "gray zone"
- Funciona porque simula comportamento normal do navegador (sem scraping agressivo)

**Distribuição do App:**
- Tecnicamente viável, mas **risco de quebra** se YouTube muda seus endpoints
- Risco baixo de takedown (Google não persegue agressivamente), mas não é garantido
- **Recomendação:** Documentar no README que usa reverse-engineering e pode quebrar

**Vantagem:** Sem necessidade de API key ou credenciais Google — mais simples para distribuição.

#### 3.3 yt-dlp

**Compliance:**
- Similar ao reverse-engineering (gray zone)
- Mais agressivo que youtube-transcript-api (simula muitos tipos de usuário)
- YouTube pode bloquear IPs de data centers facilmente

**Distribuição:**
- Risco similar ao reverse-engineering, mas maior chance de rate limit em production
- **Recomendação:** Evitar em app distribuído com múltiplos usuários

### 4. Edge Cases Identificados

#### 4.1 Vídeos Sem Legendas

**Cenário:** Usuário cola URL de vídeo sem legendas automáticas nem manuais.

**Solução:**
- Ambas as abordagens (API oficial e reverse-engineering) retornam erro claro
- Backend deve traduzir para mensagem legível: "Este vídeo não possui legendas/transcrições disponíveis"

#### 4.2 Legendas em Idioma Diferente

**Cenário:** Vídeo tem legendas em EN, não em PT-BR.

**Solução:**
- youtube-transcript-api permite especificar múltiplos idiomas com prioridade
- Backend deve tentar PT-BR, depois PT, depois EN automaticamente
- Informar ao usuário: "Transcrição disponível em [idioma], não PT-BR"

#### 4.3 Links Inválidos / Vídeos Privados / Removidos

**Solução:**
- youtube-transcript-api lança InvalidVideoId, VideoUnavailable, TranscriptsDisabled
- API v3 retorna 404 ou access denied
- Backend mapeia erros para mensagens claras no frontend

#### 4.4 Transcrições Muito Longas (Context Limit)

**Cenário:** Vídeo com 5+ horas retorna transcrição > 100k tokens.

**Solução:**
- OpenRouter chat completions tem limite de context (depende do modelo)
- Implementar truncamento no backend: se transcrição > 80k chars, avisar e truncar
- Este é um problema existente também para áudio longo — padrão já existe em 	ranscribe.rs (chunk splitting)

## Recomendações

### Recomendação Principal: youtube-transcript-api como Subprocess

**Fluxo:**
1. Usuário cola URL do YouTube
2. Backend valida URL com regex
3. Backend invoca Python subprocess: python -m youtube_transcript_api <video_id> [pt,pt-BR,en]
4. Captura stdout (JSON das transcrições) ou stderr (erro)
5. Parseia, retorna transcrição para frontend
6. Frontend continua fluxo normal: transcrição → resumo

**Por quê:**
- Simplicidade máxima (Python package já existe e é estável)
- Sem necessidade de API keys ou credenciais
- Sem dependências Rust adicionais
- Custo: Adicionar distribuição da biblioteca Python no installer do app (via pip)
- Risco: YouTube quebra reverse-engineering (improvável em curto prazo, mas possível)

**Implementação:**
- Novo command em Rust: etch_youtube_transcript(video_url: String) -> Result<String, String>
- Chamada: Command::new("python").arg("-m").arg("youtube_transcript_api").arg(video_id).output()
- Parse output como JSON
- Tratamento de erro: stderr do processo mapeado para mensagens do frontend

**Custo Estimado:** Baixo (50-80 linhas de Rust + UI simples)

---

### Recomendação Alternativa: YouTube Data API v3

Se distribuição comercial ou necessidade de compliance total com Google, migrar para:

`
1. Criar projeto em Google Cloud Console
2. Habilitar YouTube Data API v3
3. Gerar API Key (não requer OAuth para public videos)
4. Guardar API key no Tauri store (como OpenRouter key)
5. Backend: HTTP request via reqwest ao youtube.googleapis.com/youtube/v3/captions
`

**Custo Estimado:** Médio (100-150 linhas de Rust + modal de setup de API key)  
**ToS Compliance:** 100% oficial  
**Vantages:** Confiável, documentado, suporte do Google

## Riscos e Mitigation

| Risco | Severidade | Mitigation |
|-------|------------|-----------|
| YouTube quebra reverse-engineering | Média | Documentar como conhecimento limitado; planejar migração para API oficial |
| Transcrição muito longa | Baixa | Truncamento automático (já existe pattern em transcribe.rs) |
| Rate limiting em produção | Baixa | Adicionar cooldown entre requests, cache local de transcrições |
| API key vazada (API v3) | Média | Usar Tauri store (já usa AES), nunca passar para frontend |
| Vídeo privado / sem permissão | Baixa | Tratamento de erro claro, informar ao usuário |
