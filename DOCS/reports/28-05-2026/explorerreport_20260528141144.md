## Resumo Executivo

A recomendação anterior (subprocess Python com `youtube-transcript-api`) está corretamente invalidada: exige Python na máquina do usuário ou empacotamento do runtime, violando a restrição de app auto-contido para a Microsoft Store.

Reavaliando sob a nova restrição, o veredito é: **a funcionalidade é viável, mas com confiabilidade frágil e fora do controle do desenvolvedor.** A única abordagem que respeita a restrição de "sem dependências externas" é **reimplementar em Rust puro (via `reqwest`) a chamada à InnerTube API do YouTube**, idealmente reaproveitando um crate existente (`yt-transcript-rs`).

Pontos centrais que mudam a conclusão anterior:

1. **A YouTube Data API v3 oficial NÃO serve para este caso.** O método `captions.download` exige OAuth2 **e** só funciona para vídeos que o próprio usuário autenticado possui. É tecnicamente impossível baixar legendas de vídeos de terceiros pela API oficial. Esta opção deve ser descartada — não é "compliance mais rigoroso", é incapacidade funcional.
2. **A abordagem InnerTube é exatamente o que a `youtube-transcript-api` faz por baixo** — não é magia, é engenharia reversa de HTTP. Logo, é 100% reproduzível em Rust puro, sem Python, satisfazendo a restrição de auto-contido.
3. **O risco real não é distribuição nem ToS — é confiabilidade.** Em 2025/2026 o YouTube endureceu o bloqueio: PO tokens em alguns vídeos e bloqueio agressivo de IPs de datacenter. Para um app desktop instalado pelo usuário final, isso é favorável (cada usuário usa seu próprio IP residencial), mas continua sendo um endpoint não-oficial que pode quebrar sem aviso.

## 1. Rust puro via `reqwest` (InnerTube) — VIÁVEL e recomendado

### Como funciona (o mesmo que a lib Python faz)
1. POST para `https://www.youtube.com/youtubei/v1/player` com o `videoId` e um `context` de client (ex.: client web/android), com User-Agent de browser. A resposta traz os `captionTracks` com `baseUrl` de cada legenda.
2. GET no `baseUrl` da faixa desejada, acrescentando `&fmt=json3`, para obter o texto temporizado.
3. Concatenar os segmentos em texto puro → alimenta `generate_summary` sem modificação.

Não requer API key, não requer browser headless, não requer Python. É puramente HTTP + parse de JSON — exatamente o stack que o projeto já usa (`reqwest` 0.12 com feature `json`, `serde_json`, `tokio` full já estão no `Cargo.toml`).

### Crates Rust existentes
- **`yt-transcript-rs`** (akinsella) — **melhor candidato.** v0.1.8 (jun/2025), 54 commits, usa **exclusivamente a InnerTube API** (migrou para isso justamente por causa das mudanças permanentes do YouTube). Sem API key. Suporta proxy opcional e cookies. Dependências alinhadas ao projeto (`tokio`, `reqwest`). Tem `TROUBLESHOOTING.md` reconhecendo a fragilidade.
- **`ytranscript`** (rudrodip) — v0.1.0 (mai/2024), 10 commits, faz scraping de XML via regex. Praticamente sem manutenção e usa abordagem mais antiga (timedtext direto), que é a primeira a quebrar. Não recomendado.
- **`yt-dlp` (crate)** — wrapper async em torno do **executável** `yt-dlp`, reintroduz uma dependência externa binária. Descartado.

**Conclusão:** existe um crate consolidado o suficiente (`yt-transcript-rs`) que já implementa a engenharia reversa via InnerTube. A complexidade de implementação manual do zero seria Média-Alta; usar o crate reduz para Baixa.

### Decisão de dependência: crate vs. implementação manual
- **Usar `yt-transcript-rs`**: rápido (esforço baixo), mas adiciona dependência de um crate 0.1.x de mantenedor único — risco de abandono. Mitigável por ser open-source (pode-se forkar).
- **Implementar manualmente em `youtube.rs`** copiando a lógica InnerTube: ~150-250 linhas, controle total, sem dependência de terceiros. Esforço Médio.

Recomendação: começar com `yt-transcript-rs` para MVP; se o mantenedor parar ou o YouTube mudar, internalizar a lógica.

## 2. YouTube Data API v3 Oficial — DESCARTAR (incapacidade funcional)

- `captions.download` **exige OAuth2** E só baixa legendas de vídeos que **o próprio usuário autenticado possui**. Como o app resume vídeos de terceiros, isso elimina a API oficial por completo.
- Mesmo que servisse, OAuth2 num app desktop distribuído exigiria fluxo de consentimento Google por usuário, client secret embarcado (que não é secreto num app distribuído), e quota de 10.000 unidades/dia compartilhada — gargalo com múltiplos usuários.

**Não é alternativa viável.**

## 3. Outras alternativas

### APIs de terceiros (não-Google)
Existem (Apify YouTube Transcript Scraper ~US$0,50/1k vídeos, RapidAPI, etc.):
- **Prós:** confiáveis, lidam com bloqueio de IP usando proxies residenciais.
- **Contras graves:** custo recorrente por uso, exigiria chave de API paga embarcada no app (que vaza num app distribuído) ou que cada usuário criasse a sua — atrito alto.
- **Veredito:** descartar para fluxo padrão.

### WebView embarcado no Tauri
- Em tese o Tauri pode abrir uma janela WebView2 navegando até o vídeo e extrair a transcrição via JS injetado.
- **Contras:** depende do DOM do YouTube (muda com frequência), botão "Mostrar transcrição" nem sempre presente, consentimento de cookies, manutenção alta.
- **Veredito:** não recomendado como primário.

## 4. Impacto no Codebase

A etapa de resumo é genuinamente agnóstica e reutilizável sem alteração:
- `generate_summary` (`summarize.rs`) recebe `transcription: String` puro. Nada a mudar.
- O fluxo de UI converge num único estado `transcription` em `App.tsx`. Qualquer fonte que popule esse estado cai automaticamente no fluxo de resumo.

**Mudanças necessárias (mínimas):**
1. **Backend** — novo `src-tauri/src/commands/youtube.rs` com command `fetch_youtube_transcript(url: String) -> Result<String, String>`:
   - Extrair `videoId` da URL (suportar `youtube.com/watch?v=`, `youtu.be/`, `shorts/`).
   - Chamar InnerTube via `yt-transcript-rs`, preferindo idiomas `["pt-BR","pt","en"]` com fallback.
   - Mapear erros para mensagens claras: sem legendas, vídeo privado/removido, URL inválida, IP bloqueado, PO token requerido.
   - Registrar o command no builder em `lib.rs`.
2. **Frontend** — adicionar quarto `inputMode: "youtube"` ao seletor e novo componente `YoutubeUrlInput.tsx` que chama `invoke("fetch_youtube_transcript", { url })` e usa o mesmo padrão de callbacks de `FileUpload` (`onTranscriptionStart/Done/Error`/`onTextLoaded`).
3. **`Cargo.toml`** — adicionar `yt-transcript-rs`.

**Arquivos novos:** 2 (`youtube.rs`, `YoutubeUrlInput.tsx`). **Arquivos tocados:** `App.tsx`, `lib.rs`, `Cargo.toml`.

## Edge Cases

| Cenário | Tratamento |
|---------|-----------|
| Sem legendas | InnerTube retorna sem `captionTracks` → "Este vídeo não possui legendas disponíveis" |
| Idioma diferente | Fallback `pt-BR → pt → en → primeira disponível`; informar idioma usado |
| Vídeo privado/removido/restrito por idade | Erro mapeado; restrito por idade pode exigir cookies (suportado pelo crate) |
| PO token requerido (novidade 2025/2026) | Crate lança erro específico; tratar como "transcrição indisponível para este vídeo" |
| IP bloqueado (429) | Raro para usuário residencial; mensagem orientando tentar mais tarde |
| Transcrição muito longa | Mesmo limite do fluxo de áudio; tratar com aviso/truncamento (problema pré-existente) |

## Recomendação Final

**Adotar: Rust puro via InnerTube API usando `reqwest`, começando com o crate `yt-transcript-rs`.**

**Por que é a mais adequada para app distribuído na Microsoft Store:**
- 100% auto-contido: compila no binário, zero dependências externas no PC do usuário.
- Sem API key, sem OAuth, sem custo recorrente.
- Reutiliza stack já presente (`reqwest`/`serde_json`/`tokio`) e `generate_summary` intacto.
- Para app desktop, o IP residencial de cada usuário é justamente o cenário em que a InnerTube é mais confiável.

## Riscos e Mitigação

| Risco | Severidade | Mitigação |
|-------|-----------|-----------|
| YouTube muda InnerTube / quebra o fetch | Média-Alta | Isolar em `youtube.rs`; usar crate mantido; erro claro ao usuário; hotfix via atualização do app |
| Crate `yt-transcript-rs` abandonado (0.1.x, mantenedor único) | Média | Open-source: forkar/internalizar a lógica InnerTube |
| PO token / IP block em vídeos específicos | Baixa-Média (usuário residencial) | Tratamento de erro específico; não prometer 100% dos vídeos |
| ToS (engenharia reversa) | Baixa | Acesso a conteúdo público sem scraping agressivo; documentar no app a natureza não-oficial |
