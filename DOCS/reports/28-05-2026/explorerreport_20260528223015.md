# Explorer Report — Falha atual no fetch de legendas do YouTube via WebView2

> Timestamp: 20260528223015
> Task: investigar a causa mais provável da falha atual no fetch de legendas do YouTube via WebView2

## Problema
O fluxo atual em `src-tauri/src/commands/youtube.rs` consegue:
- abrir/reutilizar a WebView oculta `yt-fetch`
- carregar a watch page do YouTube
- extrair `ytInitialPlayerResponse`
- localizar URLs de legenda (`baseUrl` / `signed timedtext`)

Mas falha no passo seguinte: toda tentativa de buscar `https://www.youtube.com/api/timedtext...` volta com body vazio (`len 0`), inclusive usando a URL assinada extraída do próprio player.

## Evidências observadas

### Evidência dos logs
1. A watch page `https://www.youtube.com/watch?v=Kj79gIGkOrc` carrega corretamente com ~1.27 MB de HTML.
2. O código consegue extrair `ytInitialPlayerResponse` desse HTML.
3. O código encontra `captionTracks` / `baseUrl` de legenda.
4. Todas as tentativas para `api/timedtext` retornam body vazio (`result len: 0`).
5. Isso acontece tanto com a URL assinada quanto com os fallbacks (`fmt=json3` / `fmt=srv3`, com e sem `kind=asr`).
6. Do ponto de vista do app, o YouTube aparenta estar conectado.

### Evidência do código
1. O fetch real é feito por `webview_fetch_raw(...)`, que injeta JS com **XMLHttpRequest síncrono** dentro da WebView:
   - `x.open('GET', url, false)`
   - `x.withCredentials = true`
   - retorno é apenas `x.responseText`
2. O código **não captura**:
   - `x.status`
   - `x.statusText`
   - `x.responseURL`
   - headers de resposta
   - se houve redirect/login/challenge
3. `webview_fetch(...)` só tenta parsear o body se ele vier não vazio. Logo, o parser **não é a origem** do `len 0`.
4. `youtube_is_connected(...)` só verifica presença de cookies com nomes `SAPISID`, `__Secure-3PSID` ou `SID`. Isso prova no máximo que alguns cookies existem no store da WebView — **não prova** que a request de `timedtext` está saindo com autenticação/contexto aceitáveis para o YouTube.
5. A WebView oculta fica em `https://www.youtube.com/`, e as requests são disparadas a partir desse documento. O código **não navega a WebView para a watch page do vídeo antes** de pedir o `timedtext`.

## O que isso indica
Separando claramente:

### O que é fato
- O vídeo tem metadados de legenda visíveis na watch page.
- O problema acontece **depois** da extração das URLs.
- O problema está no **HTTP request real para o timedtext** ou no modo como o resultado dessa request chega de volta ao Rust.
- Não há evidência, pelos logs atuais, de que o endpoint esteja retornando legenda com formato inesperado; o que há é body vazio.

### O que é inferência
- O YouTube está muito provavelmente recusando, neutralizando ou esvaziando a request atual de `timedtext` por causa do **contexto da request**, não por falta de `captionTracks`.
- O bucket mais forte de falha é **transporte/request context via WebView/XHR**, com autenticação/contexto efetivo como subcausa provável.

## Hipóteses priorizadas

### 1) Mais provável — problema no transporte/request context via WebView + XHR síncrono
**Prioridade: alta**

A request atual para `timedtext` não parece equivalente ao request que o player do YouTube espera. O código usa `eval_with_callback` + XHR síncrono e só lê `responseText`. Se o YouTube responder com `204`, `403`, `status 0`, redirect para challenge/login, ou algum bloqueio específico desse fluxo, hoje tudo isso aparece simplesmente como body vazio.

Por que esta hipótese é a mais forte:
- a watch page funciona no mesmo webview
- `captionTracks` existem
- falham **tanto** a URL assinada quanto as URLs fallback
- o `len 0` acontece antes de qualquer parser

Isso aponta mais para **como** a request está sendo feita do que para erro de parsing ou ausência de legenda.

## 2) Provável — cookies/autenticação presentes no app, mas não efetivos para o timedtext
**Prioridade: média-alta**

O estado "conectado" do app é fraco como prova técnica. Ele apenas checa existência de certos cookies. Isso não garante:
- que os cookies certos estão indo na request
- que o YouTube os aceita para esse endpoint
- que a sessão está completa para esse fluxo específico

Em outras palavras: o app pode estar "conectado" do seu ponto de vista, mas a request de legenda ainda pode estar saindo sem a autenticação/contexto que o servidor espera.

## 3) Possível — headers / referer / contexto da página de origem
**Prioridade: média**

A WebView fica carregada em `https://www.youtube.com/`, não na watch page do vídeo. Mesmo sendo mesma origem, isso pode mudar o contexto efetivo da request. Se o `timedtext` hoje depender do contexto do player/watch page, o request disparado da home pode ser insuficiente.

Esse item eu trato como subcaso do problema de transporte/contexto, não como causa isolada comprovada.

## 4) Pouco provável — capability/permissões do Tauri
**Prioridade: baixa**

`src-tauri/capabilities/youtube.json` cobre as janelas `yt-login` e `yt-fetch`, e o mesmo fluxo já consegue:
- abrir a WebView
- carregar `youtube.com`
- ler cookies
- executar `eval_with_callback`
- buscar a watch page

Se capability estivesse bloqueando o essencial, a tendência seria falhar também no carregamento da watch page ou no próprio uso da WebView. Não é o que os logs mostram.

## 5) Pouco provável — arquitetura totalmente bloqueada
**Prioridade: baixa**

Os logs não mostram que WebView2 + Tauri sejam inviáveis em si. Eles mostram que a **variante atual** (XHR síncrono + retorno cego só de `responseText`) é insuficiente e muito opaca para diagnosticar o `timedtext`.

Então eu não classificaria como "arquitetura impossível". Eu classificaria como **arquitetura atual mal instrumentada e potencialmente incompatível com esse endpoint do jeito que está sendo usada**.

## O que não parece ser a causa principal
1. **Vídeo sem legendas** — contradição direta com a presença de `captionTracks` e `baseUrl` no `ytInitialPlayerResponse`.
2. **Parser XML/JSON3** — o body já chega vazio antes do parser rodar.
3. **Escolha errada entre `json3` e `srv3`** — ambos falham, inclusive a URL assinada do próprio player.
4. **Capability do `main` window** em `default.json` — o problema está nas janelas `yt-*`, e a watch page já funciona nelas.

## Conclusão
A causa mais provável da falha atual está **mais em transporte/request context via WebView/XHR do que em capability/permissões**.

Dentro desse bucket, a leitura mais forte é:
- a app consegue entrar na watch page e descobrir onde a legenda está
- mas a request real para `api/timedtext` **não está sendo aceita como uma request válida do contexto esperado pelo YouTube**
- e o código atual não coleta status/headers/redirects, então tudo colapsa para o sintoma genérico "body vazio"

Se eu tiver que escolher um diagnóstico principal em uma linha:

**o problema parece estar menos em “não há cookies” e mais em “o request atual do timedtext, feito via XHR síncrono injetado, não reproduz corretamente o contexto/autenticação efetiva que o YouTube exige”.**

## Próximos passos de validação mais úteis

### 1) Validar status/redirect/headers do `timedtext`
É o passo mais útil de todos.

Antes de tentar qualquer correção, a request precisa passar a registrar pelo menos:
- `status`
- `statusText`
- `responseURL`
- `getAllResponseHeaders()`
- tamanho do body

Isso separa rapidamente cenários como:
- `status 0`
- `204 No Content`
- `302`/redirect para login/challenge
- `403 Forbidden`
- `200` com body realmente vazio

### 2) Validar o contexto da página de origem
Fazer um teste controlado: disparar o fetch do `timedtext` **depois de navegar a WebView para a watch page do vídeo**, em vez de manter a página em `https://www.youtube.com/`.

Se isso mudar o resultado, o problema está fortemente em referer/contexto do player.

### 3) Comparar comportamento conectado vs desconectado no mesmo fluxo
Se conectado e desconectado derem o mesmo `len 0`, o estado atual de "connected" não está provando autenticação útil para o endpoint.

### 4) Validar o meio de transporte
Comparar, para uma única URL assinada fixa:
- XHR síncrono atual
- outra forma de transporte no contexto da WebView

Se o problema aparecer só no XHR síncrono, a causa fica concentrada no mecanismo atual de bridge/transporte.

### 5) Só depois disso investigar tokens extras ou exigências novas do YouTube
Ex.: contexto do player, visitor data, algum parâmetro adicional, challenge, etc.

Hoje ainda falta evidência para colocar isso em primeiro lugar.

## Dificuldade estimada da correção
**Média**

Não parece ser um problema de uma linha ou de permissão faltando. A correção provável exige pelo menos:
- melhor instrumentação da request
- validação de contexto/origem
- possivelmente troca do mecanismo atual de transporte
