### `src/components/SummaryReadView.tsx` (novo)

**Correção**
- Os 4 edge cases (vazio+loading, vazio, conteúdo+loading, conteúdo) estão cobertos e na ordem correta.
- `renderMarkdown` funciona para o esquema fixo de output do LLM (`##`, listas `- ` / `* `, parágrafos).

**Bug identificado:**
- [importante] `isListBlock` usa `lines.every(...)` — se um bloco de lista tiver uma linha em branco no meio (ex.: dois itens separados por linha vazia dentro do mesmo bloco), a linha em branco não começa com `- ` ou `* `, portanto `isListBlock` retornará `false` e o bloco será renderizado como parágrafo. Na prática o split `\n\n+` deve separar esses casos, mas itens de lista separados por apenas `\n` (sem linha em branco dupla) que não comecem com `- ` quebrarão a detecção. Risco baixo dado o prompt fixo, mas é um bug latente.

- [nit] A chave `key={i}` em elementos de lista usa o índice do bloco pai, não o índice do item. Como os blocos não são reordenados, isso é funcionalmente seguro, mas viola a recomendação do React de usar chaves estáveis. `key={j}` no `<li>` está correto.

**Simplicidade**
- [nit] `<div className="flex flex-col gap-0">` — `gap-0` é redundante (valor default). Pode ser removido.

**Efeitos colaterais**
- Componente puramente de apresentação. Sem efeitos colaterais fora do escopo.

**Padrões**
- Tipografia segue exatamente as classes definidas no planreport. Consistente com o tema escuro.
- Overlay de loading com `absolute inset-0` requer o pai com `position: relative` — satisfeito pelo `relative` no wrapper.

**Organização**
- `renderMarkdown` como função utilitária fora do componente é correto — sem acesso a estado/props.

**Problemas encontrados:** 1 importante (bug latente no parser), 1 nit.

**Bug de borda identificado:**
- [importante] `xl:border-r xl:border-r-[#1f1a36]` — em Tailwind, `border-r` define a cor usando `border-color` (shorthand), mas `border-r-[#1f1a36]` é a forma correta de colorir só a borda direita. A combinação `xl:border-r xl:border-r-[#1f1a36]` funcionará, mas a ordem importa: se `border-r` vier depois de um `border` genérico que já defina a cor, haverá conflito de especificidade. No código atual, como `border-b border-[#1f1a36]` precede, a cor `#1f1a36` já está definida pelo shorthand e `xl:border-r-[#1f1a36]` é redundante mas não prejudicial. Funciona na prática, mas poderia ser simplificado para apenas `xl:border-r border-[#1f1a36]` sem o `xl:border-r-[#1f1a36]` extra.
