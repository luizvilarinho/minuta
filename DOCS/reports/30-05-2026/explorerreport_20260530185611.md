# Explorer Report — Web Mode + Input Column Resize

**Timestamp:** 20260530185611
**Task:** Implementar modo "Web" (fetch + parse HTML) + redimensionar coluna Input e seletor de modo

---

## Descrição do problema

A task pede três alterações coordenadas:
1. Novo comando Rust `fetch_webpage_text` que faz GET em uma URL e retorna o texto legível via `html2text`
2. Novo componente React `WebUrlInput.tsx` conectado a esse comando
3. Ajustes de layout: coluna Input de `340px` → `380px` e seletor de `grid-cols-4` → `grid-cols-5`

---

## Análise técnica

### 1. `reqwest` já está presente — OK

`Cargo.toml` já declara `reqwest = { version = "0.12", features = ["json", "multipart"] }` e `tokio = { version = "1", features = ["full"] }`. Nenhuma dependência nova de runtime é necessária.

### 2. Crate `html2text` — viável, sem conflitos

Versão atual estável: **0.17.1**. A crate é **puramente síncrona** — sem dependência de tokio ou qualquer async runtime. As dependências dela (`html5ever`, `tendril`, `unicode-width`) não conflitam com `reqwest 0.12` nem com `tokio 1.x`. A função de entrada mais simples é `from_read(&bytes[..], width)` que retorna `String`.

A chamada síncrona deve ser feita dentro de `tokio::task::spawn_blocking` para não bloquear o runtime async — exatamente o mesmo padrão já usado em `youtube.rs` para o `yt-dlp`.

### 3. Padrão arquitetural — totalmente reaproveitável

`youtube.rs` serve de referência direta:
- Estrutura: validação → operação bloqueante em `spawn_blocking` → retorno `Result<String, String>`
- Decorador `#[tauri::command]` + `async`
- Registro em `mod.rs` (nova linha `pub mod web;`) e em `lib.rs` (`use commands::web::fetch_webpage_text` + entrada no `invoke_handler!`)

O novo comando é mais simples que o do YouTube: não precisa de sidecar, não cria arquivos temporários, não precisa de `TempAudio`. Apenas `reqwest::get(url).await` + `spawn_blocking` para o parse HTML.

### 4. Permissões Tauri para HTTP de saída — nenhuma configuração necessária

`capabilities/default.json` não lista permissões de rede porque o Tauri **não restringe chamadas HTTP feitas pelo backend Rust**. Restrições de `http:` no Tauri 2 se aplicam somente a chamadas HTTP iniciadas pelo **frontend** (via `fetch` do webview). O `reqwest` no backend Rust opera fora do sandbox do webview — sem nenhuma permissão adicional necessária.

### 5. Layout — 5 botões cabem com a mudança proposta

O seletor atual usa `grid-cols-4` dentro de uma coluna de `340px` com `px-8` (32px de padding em cada lado), resultando em ~276px úteis, ~69px por botão. Com os labels atuais ("Upload", "Paste", "Record", "YouTube"), o mais largo é "YouTube" com ~6 caracteres.

Com `grid-cols-5` e coluna de `380px` (~316px úteis), cada botão terá ~63px — suficiente para todos os 5 labels propostos. A label "Web" é curta, o que ajuda. A mudança de `340px` para `380px` é adequada.

Detalhe: o `inputMode` atual em `App.tsx` está tipado como `"upload" | "paste" | "record" | "youtube"`. O tipo precisa ser estendido para incluir `"webpage"`. A condição que controla visibilidade do `ModelSelector` (linha 239) também precisa ser atualizada — o modo Web **não usa** modelo de transcrição.

## Recomendações

- Adicionar `features = ["blocking"]` ao `reqwest` em `Cargo.toml` **não** é necessário — o comando será `async` com `spawn_blocking` para o parse, igual ao padrão YouTube.
- Considerar um timeout de `reqwest` (ex.: 15s) para evitar que o comando trave indefinidamente em URLs lentas. Pode ser feito com `reqwest::Client::builder().timeout(Duration::from_secs(15))`.
- A label do botão no seletor pode ser "Web" (curto) — consistente com os outros labels curtos e resolve o problema de espaço no grid.
- O componente `WebUrlInput` não precisa da prop `transcriptionModel` (diferente de `YoutubeUrlInput`) — simplifica a interface.
